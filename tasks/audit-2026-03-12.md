## 1. ROBA STUB / NON IMPLEMENTATA / HALLUCINATION

### Trovato: ZERO stub nelle strategy

Tutte e 14 le strategy usano `browse_and_extract()` e `platform_action()` reali. La lezione del 2026-03-10 ("ZERO simulation") è stata applicata. Verificato file per file:

- `freelance_writing.py` — Cerca job veri su Upwork/Fiverr, scrive contenuti, fattura
- `digital_products.py` — Usa Gumroad API vera per listare prodotti
- `micro_saas.py` — Usa Coder agent per generare MVP reali, deploya su Railway/Render
- `lead_gen.py` — Scrape vero da Google Maps, Yelp, arricchimento LinkedIn
- Tutte le altre — Stesso pattern: ricerca web reale → LLM per contenuto → delivery reale

### Trovato: 1 set di `NotImplementedError` (legittimo)

`social/api.py:39-46` — `BasePlatformClient.post()` e `get_post_metrics()` sono abstract methods. Le sottoclassi (TwitterClient, LinkedInClient, RedditClient, IndieHackersClient) implementano tutto. Questo è pattern corretto, non stub.

### Trovato: Ko-fi provider è webhook-only

`payments/kofi_provider.py` — `create_payment()`, `verify_payment()`, `get_balance()`, `send_payout()` ritornano tutti errore. Ma questo è CORRETTO: Ko-fi non ha API programmatica. Solo il webhook handler funziona. Non è stub, è limitazione della piattaforma.

### Trovato: LemonSqueezy `get_balance()` è FALSO

`payments/lemonsqueezy_provider.py` — `get_balance()` somma TUTTI gli ordini pagati ma NON sottrae: rimborsi, payout già fatti, fee della piattaforma. Il sweep engine userebbe un bilancio SBAGLIATO.

**Rischio**: Sweep potrebbe calcolare importo sweepable errato → deficit.

## 2. COSE CHE PERDONO SOLDI

### CRITICO: Refund-After-Sweep — Nessun Recovery

**File**: `payments/sweep_engine.py` + `payments/manager.py:514-519`

Se un cliente chiede rimborso DOPO che i fondi sono già stati sweeppati al creatore:
1. Il brand va in bilancio negativo
2. Il deficit viene tracciato in `sweep_deficits`
3. **Ma i soldi sono GIA' ANDATI** — nel wallet Monero del creatore
4. Non esiste meccanismo di recupero
5. Loggato come CRITICAL ma nessuna azione automatica

**Impatto**: Soldi persi. Il brand ha debito che non può ripagare.

### CRITICO: Race Condition Sweep vs Refund Concorrente

Se sweep e refund webhook arrivano contemporaneamente per lo stesso brand:
1. Sweep legge bilancio → calcola importo
2. Refund processa → riduce bilancio
3. Sweep invia vecchio importo → usa soldi già rimborsati
4. Risultato: deficit irrecuperabile

**Nessun lock** tra sweep e webhook processing.

### CRITICO: Webhook Handler Ritorna 200 OK su Failure

**File**: `payments/webhook_server.py:266-271`

```python
for handler in self._event_handlers:
    try:
        await handler(event)
    except Exception as e:
        logger.error(...)
        # MA: nessun re-raise → webhook marcato come processato!
```

Se il handler crasha, il webhook è considerato processato con successo. Il provider non riprova. **Pagamento perso silenziosamente.**

### ALTO: GL Recording Fuori dalla Transaction

**File**: `payments/manager.py:419-420`

Il GL entry viene registrato DOPO la transaction del webhook:
1. Webhook processato e salvato ✓
2. GL entry fallisce ✗
3. Webhook marcato come processato ma GL non aggiornato
4. Riconciliazione trova discrepanza ma soldi già contati

### ALTO: Monero Fee Estimation Fallback

**File**: `payments/monero_provider.py:150-152`

Se la stima fee RPC fallisce → fallback a 0.00005 XMR hardcodato. Se la fee reale è più alta, la transaction fallisce mid-broadcast. Peggio: tra stima e invio il balance può cambiare (race).

### ALTO: Monero Send Senza Conferme

**File**: `payments/monero_provider.py:314`

`send_payout()` ritorna PENDING immediatamente. Nessun wait per conferme blockchain. Se il wallet si spegne prima del broadcast, TX persa ma sweep_id marcato come completato.

### MEDIO: Rate Limiter Webhook con Race Condition

**File**: `payments/webhook_server.py:81-93`

Il rate limiter non ha lock. Request concorrenti possono manipolare i contatori contemporaneamente. Inoltre quando il limite viene raggiunto, TUTTO il dizionario IP viene cancellato.

### MEDIO: Pagamenti Droppati Silenziosamente

**File**: `payments/manager.py:307-319`

Se amount < 0, > €1M, o NaN → `return` silenzioso. Nessuna eccezione, nessuna coda retry, nessun log nel DB. Il webhook provider pensa che sia andato bene (200 OK dal server).

## 3. COSE RISCHIOSE / PERICOLOSE

### CRITICO: Social Media Posting Autonomo LIVE

**File**: `strategies/social_media.py:220`

```python
result = api_client.post(full_content)
```

Questo **POSTA REALMENTE** su Twitter, LinkedIn, Reddit. Nessun gate di conferma umana. Se il contenuto è sbagliato, offensivo, o fuori target → danno reputazionale immediato al client e al brand.

Anche `social/api.py` conferma: TwitterClient, LinkedInClient, RedditClient fanno POST reali alle API delle piattaforme.

### CRITICO: Monero send_payout() Invia XMR Reali

**File**: `payments/monero_provider.py:285-290`

Chiama `transfer` RPC con `ring_size=16`. Non c'è simulazione, non c'è dry-run. Se l'indirizzo è sbagliato o compromesso → XMR persi per sempre (irreversibile su blockchain).

### ALTO: BTCPay send_payout() Invia BTC Reali

**File**: `payments/btcpay_provider.py:250-254`

Crea transazioni Bitcoin on-chain. Irreversibile.

### ALTO: Email Marketing Senza Conferma

**File**: `business/email_marketing.py:129-146`

Campagne email inviate senza:
- Approvazione umana
- Rate limiting
- Verifica unsubscribe prima dell'invio
- Review del contenuto

### ALTO: Proxy Non Enforced su Tutti i Provider

**File**: `payments/stripe_provider.py:45-70`

Il proxy è opzionale. Se `proxy_url` non è configurato o auto-detect fallisce → connessione DIRETTA. Il vero IP del creatore viene esposto a Stripe, Gumroad, etc.

**File**: `payments/base.py:35-36` — se auto-detect fallisce, proxy diventa stringa vuota.

### MEDIO: Webhook Replay Cancella Idempotency

**File**: `payments/manager.py:638-643`

`replay_webhook()` cancella i record di idempotency per forzare il reprocessing. Se chiamato per errore → duplicate payment processing.

### MEDIO: Database Senza Lock Concorrente

**File**: `db/database.py:126-139`

SQLite WAL mode ma nessun lock applicativo. Operazioni finanziarie concorrenti possono leggere balance stale.

### Cose che mancano per "vivere da solo":

1. **MAI testato con soldi veri** — 1,520 test ma tutti mock/unit
2. **Nessun ambiente staging** — si passa direttamente da test a produzione
3. **Nessun monitoring in tempo reale** — dashboard SSE c'è ma nessun PagerDuty/alerting esterno
4. **Nessun dead letter queue** per webhook falliti — persi per sempre
5. **Nessun circuit breaker** — se Monero RPC va giù, sweep loop si blocca indefinitamente
6. **Config encryption key** in `~/.monai/.config_key` — se permissions sbagliate, password leggibili

## 5. RACCOMANDAZIONI PRIORITARIE

### Must-Fix Prima del Deploy (BLOCCANTI)

1. **Wrap webhook handler in try/except che re-raise** — MAI ritornare 200 OK se il processing è fallito
2. **Lock brand durante sweep** — impedire webhook concorrenti di modificare balance durante un sweep
3. **Dead letter queue per webhook** — webhook falliti devono essere salvati e riprovabili
4. **Timeout per-call su Monero RPC** — `asyncio.wait_for()` su ogni chiamata RPC
5. **Conferma umana (Telegram) per sweep > €50** — il creatore approva prima dell'invio crypto
6. **Fix LemonSqueezy get_balance()** — deve sottrarre rimborsi e payout già fatti
7. **Enforce proxy su TUTTI i provider** — se proxy non disponibile, BLOCCA l'operazione
8. **Conferma umana per social media posting** — draft → review Telegram → approvazione → post

### Should-Fix (Importanti ma non bloccanti)

9. Rate limiting su sweep (max 1/ora per brand)
10. Validazione Decimal su tutte le conversioni amount (no float)
11. Schema validation su webhook deserialization
12. Retry con backoff su Stripe 429
13. Verifica backup riuscito prima di permettere file deletion
14. Circuit breaker su external API calls

## 6. CONCLUSIONE

Il sistema è **architetturalmente maturo** — buon design, buoni test, buone lezioni apprese. Le strategy NON sono hallucinate, usano dati reali. I payment provider sono integrazioni vere.

**MA**: il plumbing finanziario ha buchi che in produzione PERDERANNO soldi su edge case (refund-after-sweep, webhook failure, race condition). Il social media posting autonomo è un rischio reputazionale serio.

**Prima di deployare con soldi veri**: fixare i 8 punti bloccanti sopra, poi fare un test end-to-end con micro-importi (€1-5) su Ko-fi per validare l'intero flusso webhook → GL → sweep → creatore.
